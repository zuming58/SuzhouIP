## 1. 项目当前目标

苏丽娘项目当前方向是做一个苏州文旅 AI 数字导游：

1. 用户通过网页中的数字人界面与“苏丽娘”对话。
2. 先用火山引擎豆包 Realtime API 打通真人级实时语音对话。
3. 后续再把用户意图提炼出来，调用项目自己的业务能力，例如生成路线、点位讲解、附近美食推荐、加入行程。
4. 最终形成比赛演示闭环：欢迎页 -> 主界面 -> 语音问答 -> 路线推荐 -> 导览讲解 -> 附近服务 -> 我的行程。

当前重点是：**实时语音已经基本跑通，但数字人视频状态切换仍不稳定，需要下午优先重构。**

### 2.4 火山 Realtime 语音桥接

位置：`voice_bridge/`

已经实现的方向：

- 浏览器前端不直接连接火山 Realtime。
- 前端只连本地 `voice_bridge`。
- `voice_bridge` 再连接火山 Realtime WebSocket。
- 这样可以避免把火山密钥暴露在浏览器里。

当前语音链路：

1. 用户在语音页按住麦克风按钮。
2. 浏览器请求电脑麦克风权限。
3. 前端使用 Web Audio API 采集声音。
4. 前端把音频转成 16k 单声道 PCM int16 little-endian。
5. 前端按小包发送给本地 `voice_bridge`。
6. `voice_bridge` 转发给火山 Realtime。
7. 火山返回 ASR 文本、模型回答文本、TTS 音频流。
8. 前端展示识别文本和回答文本。
9. 前端播放火山返回的 TTS 音频。

当前语音能力状态：

- 文本测试按钮可以触发真实链路。
- 按住说话的语音输入已经可以识别。
- 火山返回的回答可以展示。
- TTS 播放已经能听到。
- 页面右上角可以显示 `真实`，代表当前不是 Mock。

## 5. 当前最重要的问题点

### 问题 1：右侧状态按钮没有接到语音小窗口

现象：

- 在语音页右侧点击 `idle / listen / think / speak`。
- 中间语音小窗口不变化。
- 看起来像“不管按什么都是一个状态”。

初步定位：

- 右侧按钮有 `data-video`。
- 当前点击处理里只调用了 `setAvatar(...)`。
- `setAvatar(...)` 控制的是全局数字人视频，不是语音页的 `#voice-orb` 小窗口。
- 语音小窗口需要走 `requestVoiceState(...)` 或专门的 `setVoiceVideo(...)`。

相关位置：

- `suliniang_ui_demo/app.js`
- 事件处理大约在 `document.addEventListener("click", ...)` 里。
- 需要检查 `data-video` 按钮逻辑。

### 问题 2：视频切换函数不够稳定

现象：

- 有时切换状态没有生效。
- 有时出现白框。
- 有时一直停在旧状态。
- 有时人物突然消失。

初步原因：

- 语音页用了双 `<video>` 交叉切换。
- 当前逻辑会等下一个视频 `readyState` 和 `videoWidth` 准备好再切换。
- 如果某个时间点还没准备好，函数可能直接返回。
- 返回后没有可靠的重试机制。
- 于是状态已经更新，但画面没切过去。

下午建议：

- 不要继续在旧函数上加零散补丁。
- 重写一个更简单的状态机：
  - 只有一个当前状态。
  - 只有一个目标状态。
  - 只有一个切换锁。
  - 视频没加载好时继续保留旧视频，不显示白框。
  - 新视频确认可播放后再切换 active。
  - 切换失败时回退到 `idle_loop.mp4`。

### 问题 3：状态优先级和延迟规则不清晰

现象：

- 轻触一下按钮也触发状态变化。
- 用户还没真正讲话就切倾听。
- AI 还没说完就回待机。
- 说完后又没有回待机。
- 状态之间抢来抢去。

需要重新定义状态规则。

建议状态优先级：

1. `speaking`：最高优先级，只要火山开始返回 TTS 音频或回答文本，就切说话。
2. `listening`：用户按住按钮且超过最小按住时间后，切倾听。
3. `thinking`：用户松开按钮且有有效音频后，切思考。
4. `idle`：没有录音、没有识别、没有回答、没有播放时，回待机。

建议误触规则：

- 按住小于 `450ms`：视为误触，不发送，不切长期状态。
- 音频字节量太少：视为无效输入，不发送或结束会话。
- 按住开始后可立即让按钮变红，但视频可以延迟 `150-200ms` 再切倾听，避免误触闪烁。

建议回待机规则：

- `tts.start` 或 `chat.partial` 后进入 `speaking`。
- `tts.end` 后不要立刻回 `idle`，需要等实际音频播放队列结束。
- 播放队列结束后延迟 `500-800ms` 回 `idle`。
- 如果超过兜底时间，例如 `20-24s`，强制回 `idle`，避免卡死在说话。

### 问题 4：人物裁切位置仍需微调

用户反馈：

- 放大方向是对的。
- 小窗口里不需要显示完整人物，可以显示到膝盖以上。
- 但现在头部太靠上，头已经跑到外面。
- 需要整体往下移一点。
- 视频背景和矩形框有色差，所以放大一点是合理的，可以遮住边缘。

相关 CSS：

- `suliniang_ui_demo/styles.css`
- `.voice-orb`
- `.voice-video`

当前思路：

- 保持放大。
- 小窗口可以稍微拉宽。
- 通过 `object-position` 或 `transform: translateY(...) scale(...)` 把人物往下挪。
- 先保证四个核心视频同样大小、同样位置，再微调视觉。

## 7. 下午建议实现方案

### 7.1 不要继续混用 `setAvatar` 和 `setVoiceVideo`

当前混乱的根源之一是：

- `setAvatar()` 控制全局数字人视频。
- `setVoiceVideo()` 控制语音页小窗口。
- 右侧按钮、语音事件、页面事件有时走不同函数。

建议下午明确拆分：

- 全局页面数字人：继续使用 `setAvatar()`。
- 语音页小窗口：只允许使用 `setVoiceAvatarState(state, reason)`。

新增或重写一个统一入口：

```js
setVoiceAvatarState("idle", "page-enter");
setVoiceAvatarState("listening", "press-confirmed");
setVoiceAvatarState("thinking", "audio-end");
setVoiceAvatarState("speaking", "tts-start");
```

外部不要直接调用 `setVoiceVideo()`。

### 7.2 语音状态映射固定

```js
const voiceAvatarStates = {
  idle: {
    video: "idle_loop.mp4",
    label: "可以继续追问",
    priority: 0,
  },
  listening: {
    video: "listening_loop.mp4",
    label: "正在听，松开后发送",
    priority: 1,
  },
  thinking: {
    video: "thinking_loop.mp4",
    label: "苏丽娘正在思考",
    priority: 2,
  },
  speaking: {
    video: "speaking_loop.mp4",
    label: "苏丽娘正在播报",
    priority: 3,
  },
};
```

### 7.3 视频切换要有“旧视频兜底”

切换原则：

- 当前视频继续显示。
- 预备视频设置新 `src`。
- 等新视频 `canplay` 或 `loadeddata`。
- 新视频播放成功后才加 `active`。
- 新视频没准备好时，不隐藏旧视频。
- 如果 1 秒还没准备好，保留旧视频并记录状态，不白屏。

### 7.4 不要让一次性动作进入主语音状态

语音主状态只用四个 loop：

- `idle_loop.mp4`
- `listening_loop.mp4`
- `thinking_loop.mp4`
- `speaking_loop.mp4`

`guide_once.mp4`、`smile_once.mp4`、`recital_once.mp4` 不参与语音主状态切换。

### 7.5 右侧按钮用于测试语音小窗口

右侧“数字人状态”按钮在语音页应该能直接测试小窗口：

- 点击 `idle` -> 小窗口切 `idle_loop.mp4`
- 点击 `listen` -> 小窗口切 `listening_loop.mp4`
- 点击 `think` -> 小窗口切 `thinking_loop.mp4`
- 点击 `speak` -> 小窗口切 `speaking_loop.mp4`

如果当前页面不是语音页，可以继续控制全局数字人。
